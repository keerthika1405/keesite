Lock

A MySQL lock is a flag that can be assigned to a table to alter its properties. A client-server can assign a table lock to prevent other sessions from accessing the same table during a specific time frame.
READ LOCK: Allows a user to only read the data from a table
WRITE LOCK: Allows a user to do both reading and writing into a table
A client session can acquire or release table locks only for itself. A locked table remains locked until you either commit your transaction or roll it back

a primary key is a column or combination of columns that uniquely identifies each record in a database table.

Foreign Key

A FOREIGN KEY is a field (or collection of fields) in one table, that refers to the PRIMARY KEY in another table. The table with the foreign key is called the child table, and the table with the primary key is called the referenced or parent table.

          FOREIGN KEY (PersonID) REFERENCES Persons(PersonID); -- foreign key
          CONSTRAINT FK_PersonOrder FOREIGN KEY (PersonID) REFERENCES Persons(PersonID)
          ALTER TABLE Orders ADD FOREIGN KEY (PersonID) REFERENCES Persons(PersonID);
          ALTER TABLE Orders ADD CONSTRAINT FK_PersonOrder FOREIGN KEY (PersonID) REFERENCES Persons(PersonID);
          ALTER TABLE Orders DROP FOREIGN KEY FK_PersonOrder;



Composite Key

A Composite Key can be created by combining more than one Candidate Key. Each Candidate Key (or column) that makes up a Composite Key may or may not be a Foreign Key. However, if all the columns of the Composite Key are Foreign Keys in their own right, then the Composite Key is known as a Compound Key. A Composite Key cannot be NULL; i.e. any column of the Composite Key must not contain NULL values. The individual columns making up the Composite Key can contain duplicate values, but, the combination of these columns must be unique across the database table.




Unique Key

The unique key guarantees the uniqueness of the columns in the database. It is similar to the primary key but can accept a null value




Alternate Key

an alternate key is a set of candidate keys that are not selected as the primary key. Alternate keys are also known as secondary keys. They can be used to uniquely identify a record in a table. A table can have multiple candidate keys, but only one can be the primary key. The remaining candidate keys are called alternate keys.




Super Key

A Super key is a single key or a group of multiple keys that can uniquely identify tuples in a table

          {Emp_SSN }
          { Emp_Id }
          { Emp_email }
          {Emp_SSN, Emp_Id }
          { Emp_Id, Emp_name}
          {Emp_SSN, Emp_Id, Emp_email }

Candidate Key

Candidate key is a subset of Super keys and devoid of unnecessary attributes that are not important for uniquely identifying tuples.

Outer Joins

An outer join is a type of SQL join that retrieves records from one table even if there are no corresponding records in another table. There are three main types of outer joins: LEFT OUTER JOIN, RIGHT OUTER JOIN, and FULL OUTER JOIN.

LEFT OUTER JOIN

Returns all records from the left table and the matched records from the right table. If there is no match in the right table, NULL values are returned for columns from the right table.

RIGHT OUTER JOIN

Returns all records from the right table and the matched records from the left table. If there is no match in the left table, NULL values are returned for columns from the left table.

FULL OUTER JOIN (full join)

While the term "FULL JOIN" is commonly used, MySQL, for example, doesn't directly support it. You can achieve a full outer join using a combination of left and right joins with the UNION operator.




FULL JOIN ( full outer join)

While the term "FULL JOIN" is commonly used, MySQL, for example, doesn't directly support it. You can achieve a full outer join using a combination of left and right joins with the UNION operator.




Cross Joins

Returns the Cartesian product of both tables, i.e., all possible combinations of rows.




Left Joins

Returns all rows from the left table and the matching rows from the right table. If there is no match, NULL values are returned for columns from the right table




Right Joins

Returns all rows from the right table and the matching rows from the left table. If there is no match, NULL values are returned for columns from the left table.




Self Joins/Cartesian Join

A SELF JOIN is a specific case of a SQL join where a table is joined with itself. This can be useful when you want to compare rows within the same table. When you perform a SELF JOIN, you typically use aliases to distinguish between the different instances of the same table.




clause

In MySQL, a clause is a component of a SQL statement that provides additional instructions to the database server. There are several types of clauses in MySQL, each serving a specific purpose within different SQL statements.




WHERE Clause

Filters the rows to be retrieved based on a specified condition in the SELECT, UPDATE, or DELETE statements.




USING Clause

The USING clause in MySQL is used in the context of JOIN operations, specifically for specifying the columns used for joining two tables when those columns have the same name in both tables.




HAVING Clause

Filters the results of a GROUP BY query based on a specified condition.




Distinct Clause

In MySQL, the DISTINCT keyword is used in the SELECT statement to retrieve unique values from a specified column or combination of columns. It eliminates duplicate rows from the result set, and only one occurrence of each distinct combination is included in the output.




LIMIT Clause

In MySQL, the LIMIT clause is used to restrict the number of rows returned by a SELECT statement. This can be helpful when you want to retrieve only a specified number of records from the result set.




OFFSET Clause

OFFSET clause is used in conjunction with the LIMIT clause in MySQL to skip a specified number of rows before returning the result set. It's often used for implementing pagination.




Union Clause

The UNION clause in MySQL is used to combine the result sets of two or more SELECT statements. It removes duplicate rows from the combined result set.

Atomicity: (ALL or Nothing) when you mark any sequence of methods call as transactional
they all must execute (successfully as one) or (fail as one)
Any failure in the chain should trigger an undo of the call before the failure point as well as RollingBack any data change made prior to failure [Atomicity in a database transaction means that all the changes made during that transaction are treated as one "bundle" of changes when you are trying to modify your database, it's either all of the changes happen at the same time, or none of them happen at all.] Consistency: a database insert must respect integrity rules and constraints [Consistency in databases means that the data stored in the database is always in a valid and consistent state. For example, if the database contains any constraints such as primary keys, foreign keys, and so on, it should always conform to the rules surrounding the constraint. let's say a table has a rule that says a specific column must be an integer value. Consistency ensures that this rule is always followed and data inserted into the column can only be of the integer value data type. ] Isolation: here is where multithreading really comes in isolation is guarantee that you interaction with database is protected from any adverse effect of concurrent access to same data [The ability of multiple transactions to execute without interfering with one another is known as Isolation. The isolation level of a transaction determines how the changes made by that transaction are visible to other transactions.] Durability durability is a guarantee of assets that stipulate as long as the transaction has been committed successfully to the database , the data stay committed even in the event of a massive system failure (like disk or hardware kind of failure)
