  11. Dynamic Binding
    Dynamic binding is also referred to as a run-time polymorphism. In this type of binding, the functionality of the method call is not decided at compile-time. In other words, it is not possible to decide which piece of code will be executed as a result of a method call at compile-time. Overriding a method is a perfect example of dynamic binding in java. 
    
  class India {

    public void locate() {
      System.out.println("We are in India");
    }
  }
  
  class SubIndia extends India {
  
    @Override
    public void locate() {
      System.out.println("We are in Delhi");
    }
  }
  
  public class DynamicBindingTest {
  
    public static void main(String args[]) {
      // Method of super class or parent class is called
      India superobj = new India();
      superobj.locate();
      
      // Method of sub class or child class is called by Parent reference, 
      // this is called "Dynamic Binding"
      India subobj = new SubIndia(); // upcasting
      subobj.locate();
  
      SubIndia subobj2 = new SubIndia();
      subobj2.locate(); // method of sub class or child class is called
    }
  }
output:
We are in India
We are in Delhi
We are in Delhi

In this code, there is a superclass called India that has a method called locate(), which simply prints "We are in India".

     The subclass SubIndia extends India and overrides the locate() method to print "We are in Delhi".
    
In the main() method, an object of India is created and its locate() method is called, which prints "We are in India".
Then, an object of SubIndia is created and upcast to the type of India. When the locate() method is called on this object using the India reference variable,
    
dynamic binding comes into play and the overridden locate() method in the SubIndia class is called, printing "We are in Delhi".
    
This is because the actual type of the object is SubIndia, and so its implementation of the locate() method is used.
    
Finally, an object of SubIndia is created directly and its locate() method is called, which again prints "We are in Delhi".

Types of Garbage Collectors in Java
There are basically four types of Garbage collector

Serial GC
Parallel GC
CMS GC
G1 GC
1. Serial GC
This GC runs with a single thread, in other words, when this GC runs it freezes all other threads and does its job. Hence it is not an ideal solution for a multi-threaded application.

2. Parallel GC
This GC works in a multi-threaded manner. It runs multiple threads to collect the garbage, but like the previous one, it also stops other threads while running GC threads. It is the default GC of the JVM, and the number of threads, pause time, etc can be controlled through the command line argument. If not configured properly, this may have an impact on the overall performance of the application.

3. CMS GC
Concurrent Mark Sweep, known as CMS garbage collector, this collector also works in multiple GC threads but this doesnâ€™t stop the application threads while itself is running. This collector is generally used in an application with a large set of long-lived data, and the hardware has two or more processors which will benefit in running GC threads along with the application threads.

4. G1 GC
Garbage First, known as G1 Garbage collector is also a multi-threaded GC. This is the successor of CMS GC and is available after Java 7. This is a more optimized and powerful GC and is preferably used in multiprocessor machines that have large heap memory. This works in two phases, first, it marks the liveness of objects in heap, and when it collects all the information of empty spaces in heap, it performs the cleanup activity.

JDK is a software development kit whereas JRE is a software bundle that allows Java program to run, whereas JVM is an environment for executing bytecode.

Usually, it's any operating system, for example, Unix, Linux, Microsoft Windows, or macOS. Here JRE acts as a translater and facilitator between the java program and the operating system..

When you run a program using the JRE and JDK, it also goes to the JVM as it is required to run the java program and interprets the program. The phases are as follows: Compile the Code: The Java Development Kit(JDK) provides us with the JAVAC compiler to get through this step. We basically require the JDK to convert our source code into a specific format (compiled code) that can be easily interpreted by the Java Runtime Environment(JRE). Run the Code: JVM runs the bytecode provided by the compiler. Since Java is a platform-independent language, the compiled code produced by the JAVAC compiler is converted to machine code using platform specific JVMs. Different platforms have different JVMs. JVMs convert the bytecode into platform specific machine code.

    The Java Virtual Machine has a method area that is shared among all Java Virtual Machine threads. You may think that this area stores all the compiled code. It stores per-class structures such as the run-time constant pool, field and method data, and the code for methods and constructors.     35. How does the "HashMap" work in Java?
    HashMap in Java is a part of the Java Collections Framework and is used to store key-value pairs. It provides efficient and constant-time complexity for basic operations like put, get, and remove.
    • HashMap uses an array of linked nodes or entries to store key-value pairs. This array is called the "table," and each element in the array is a "bucket."
    • The hash code is then used to determine the index (bucket) where the key-value pair should be stored in the array
    • Each node represents a key-value pair and typically includes fields for the key, value, hash code, and a reference to the next node in the linked list
    • Hash collisions occur when two different keys produce the same hash code. To handle collisions, each bucket in the array can store a linked list (or, in later versions, a tree) of entries.
    • HashMap automatically resizes itself when the number of entries exceeds a certain threshold. This process is called "rehashing." During rehashing, the size of the array (number of buckets) is doubled, and all entries are redistributed to new buckets based on their updated hash codes.
    • The "load factor" is a parameter that determines when the HashMap should be resized. The default load factor is 0.75, meaning that the HashMap is resized when it's 75% full. A higher load factor reduces memory overhead but increases the likelihood of collisions

    47. Explain the "Executor" framework in Java.
    The Executor framework in Java provides a higher-level replacement for traditional thread management. It is part of the java.util.concurrent package and is designed to simplify the task of managing and controlling threads in a concurrent program. The primary interface in the Executor framework is the Executor interface, and there are several concrete implementations and extensions available.
    They offer several advantages and address common challenges associated with direct thread management.

    why we need executor:
    • Executors abstract away the details of thread creation, scheduling, and lifecycle management. This abstraction makes it easier to focus on the logic of your tasks rather than the intricacies of thread management.
    • Executors provide efficient resource management by reusing threads from a thread pool. This helps avoid the overhead of creating and destroying threads for every task, which can be expensive.
    • Thread pools, provided by implementations like ThreadPoolExecutor, allow you to reuse a fixed number of threads to execute multiple tasks. This is especially useful when dealing with a large number of short-lived tasks, as it avoids the overhead of creating new threads for each task.
    • Executors offer mechanisms for controlling the level of concurrency in your application. For example, you can configure the number of threads in a pool or set limits on the number of tasks that can be submitted concurrently.
    • Executor frameworks provide mechanisms for handling errors that occur during task execution. They also offer features for monitoring the progress of tasks, tracking completion, and handling exceptions.
    • Executors provide a scalable solution for concurrent programming. By allowing you to control the number of threads and manage workloads efficiently, they contribute to building scalable applications.
    • Executors support graceful shutdown of the thread pool, allowing in-flight tasks to complete while preventing the submission of new tasks. This is important for ensuring that the application can shut down cleanly.

    48. What is a deadlock in Java? How do you avoid it?
    A deadlock in Java occurs when two or more threads are blocked forever, each waiting for the other to release a lock. In other words, it's a situation where a set of threads is unable to proceed because each is holding a lock and waiting for another thread in the set to release a lock. A deadlock typically involves two or more threads and multiple locks. It can happen when the following four conditions are met:
    • mutual Exclusion: At least one resource must be held in a non-sharable mode (exclusive control), meaning only one thread can use it at a time.
    • Hold and Wait: A thread must be holding at least one resource and waiting to acquire additional resources held by other threads
    • No Preemption: Resources cannot be forcibly taken away from the thread holding them. The only way a thread can release resources is voluntarily.
    • Circular Wait: A circular chain of two or more threads, each holding a resource and waiting for another resource acquired by a different thread in the chain.
